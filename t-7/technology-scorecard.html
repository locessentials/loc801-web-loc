TOPIC 7

Translation Technology Scorecard: Making an Informed Choice

A focused evaluation framework for assessing Translation Environment Tools (TEnTs) through the lens of website localization—covering web-specific file handling, team collaboration, terminology consistency, and translation quality.

🎯 Learning Objectives

  • Identify the TEnT features that matter most in a website localization context
  • Evaluate collaboration and workflow features that support distributed translation teams
  • Assess terminology module design and its impact on brand and content consistency
  • Compare the machine translation and LLM integration quality of different tools
  • Apply a structured scoring process to compare two or more TEnTs side by side

⚠️ Before You Reach for a CAT Tool

Let's be direct: for most website copy, we do not recommend using a Computer-Assisted Translation (CAT) tool—also called a Translation Environment Tool, or TEnT.

💡
Why not? Website copy—especially landing pages, calls to action, and blog posts—is written to persuade, engage, and convert. It relies on voice, rhythm, and cultural resonance. TEnTs are optimized for translating documents and technical content where consistency and efficiency take priority over creativity. When you run website copy through a TEnT, you often get output that is technically accurate but tonally flat. Your localized site deserves better.

That said, there are legitimate use cases for TEnTs in a web localization workflow. If your site includes large volumes of structured, technical content—help documentation, product specifications, support FAQs, or UI strings across a complex application—a TEnT can provide real efficiency and consistency gains.

If you've assessed your project and decided that a TEnT is the right tool, this scorecard will help you choose the right one. Work through each section and score the tools you're evaluating. At the end, compare your results.

📋 How to Use This Scorecard

🗂️
Test before you score. Spend at least two to four hours working in each TEnT with real content from your site—ideally a mix of UI strings, a help article and content like product specifications. Scores based on demos or videos will not reflect the actual experience. Use the same source content across all tools you're comparing so results are meaningful.

The scorecard is organized into seven evaluation areas. Each area includes specific questions or criteria, a rating scale, and space for notes. Rate each criterion, then calculate a section score. The final section gives you a weighted total to help compare tools head to head.

📊 The Scorecard

🌐

1. Web-Specific File Format & Content Support

Most TEnTs designed for document translation are also configured to process the file types that make up a modern website as well. Still, before committing to any tool, verify that it handles your actual content types without data loss or formatting damage.

🌐 Why this matters for website localization Websites are built from structured content: HTML files, JSON and YAML resource strings, WordPress or CMS exports, JavaScript localization files, and SVG graphics with embedded text. A TEnT that can't parse these formats correctly will either strip out content, break the file structure, or force translators to work around the tool—defeating the purpose of using one.

Criteria to Evaluate

  • Supports HTML and can correctly identify translatable text without exposing raw tags to translators
  • Handles JSON, YAML, and .po (gettext) file formats for UI strings and CMS exports
  • Preserves inline formatting (bold, links, variables, placeholders) through the translation workflow
  • Correctly processes XML-based content including SVG files with translatable text
  • Offers a live preview or side-by-side view so translators can see strings in context, not just as isolated segments
  • Handles character limits and string length warnings for UI elements
🤝

2. Collaboration & Workflow Features

Website localization is rarely a solo endeavor. Even a small project typically involves at least one translator, a reviewer, and a project coordinator. Larger projects may include multiple translators, subject matter reviewers, in-country reviewers, and client approvers. A TEnT's collaboration infrastructure can make the difference between a smooth, accountable workflow and a confusing chain of emailed files.

🌐 Why this matters for website localization Website content changes frequently. New product launches, seasonal campaigns, and regulatory updates generate revision cycles. A TEnT that supports structured workflows, version history, and clear communication between team members reduces the risk of publishing stale or inconsistent translations.

Criteria to Evaluate

  • Supports role-based access (translator, reviewer, project manager, client) with appropriate permissions for each
  • Allows real-time or near-real-time collaboration so multiple people can work in the same project simultaneously
  • Provides a notification system that alerts team members when tasks are assigned, completed, or need attention
  • Includes a comment system that allows translators and reviewers to communicate at the segment level—not just via email outside the tool
  • Comments can be attached to individual segments and can pass through workflow stages (translation → review → approval) so context is never lost
  • When a segment is recycled from translation memory, any comments previously attached to that segment are surfaced—so reviewers know the history and reasoning behind a translation choice
  • Supports change tracking and revision history so you can see what was changed, by whom, and when
  • Allows project managers to track progress at both the file and segment level without having to interrupt translators to request status updates
📖

3. Terminology Management

Terminology consistency is a brand and usability issue. If your product is called a "dashboard" in English, the localized version should use whatever term was agreed upon for that locale—every time, across every page. A well-designed terminology module enforces consistency at the moment of translation, not after the fact during QA.

🌐 Why this matters for website localization For websites, the stakes of terminology inconsistency are high. Search engines index individual pages, and if your SEO keyword is "gestión de proyectos" on one page and "administración de proyectos" on another, you split your relevance signal. Your UI becomes confusing if the same button says two different things on two pages. And your brand voice suffers if product names and taglines are translated inconsistently across the site.

Criteria to Evaluate

  • Terminology entries are concept-based, not just word-based—each entry can carry a definition, usage note, and context
  • Forbidden terms and preferred alternatives can be specified per locale and per subject domain
  • The tool highlights or flags recognized terms in the translation editor in real time, so translators don't have to remember to check
  • Automated QA can check for missing or incorrect term usage and flag segments before delivery
  • Termbase can be shared across projects, so consistent terminology carries forward as a project grows or more languages are added
  • Import and export options exist (ideally compatible with TBX, the standard interchange format) so you can build on existing terminology assets
  • Entries can include grammatical metadata relevant to the target language (gender, plural forms, declensions), which matters especially for heavily inflected languages
  • The interface for maintaining the termbase is usable by someone who isn't a translation engineer—so your client or in-house reviewer can update it without special training
🤖

4. Machine Translation & LLM Integration Quality

Most TEnTs now offer integration with one or more machine translation (MT) engines and, increasingly, with large language models (LLMs). MT and LLM output can serve as a drafting aid for human translators—but the quality of that draft varies significantly depending on the language pair, the domain, and the engine. Evaluating this properly requires testing with your actual content and language pairs.

🌐 A critical caveat: LLMs are trained primarily in English Most large language models—including the most widely used commercial ones—were trained on corpora that are overwhelmingly English-language. Their fluency in other languages varies considerably. Some perform well in major European languages; performance often drops sharply for less-resourced languages. More importantly, LLMs tend to produce fluent-sounding output even when it's wrong—which makes errors harder to catch. Never use LLM-generated translations without review by a qualified human translator in the target language.

Criteria to Evaluate

  • The tool offers access to multiple MT engines (e.g., DeepL, Google Translate, Microsoft Translator) so you can select the engine that produces the best quality a specific target language
  • LLM integration options are available (e.g., GPT models, Gemini, DeepSeek) for languages where LLMs may outperform traditional MT
  • MT and LLM suggestions are clearly marked as machine-generated so translators know which segments need what kind of verifications
  • The tool supports adaptive MT: the system learns from post-editing corrections made during the project and improves subsequent suggestions
  • You can customize or fine-tune an MT engine using your own corpora and terminology to get translations that better reflect your style, domain, and voice
  • MT and LLM integration can be disabled at the project or file level for sensitive content where machine pre-translation is not appropriate
  • There is transparency about how third-party MT and LLM providers handle your data—especially important if your content includes pricing, proprietary product descriptions, or legally sensitive language
  • Usage analytics (API calls, volume, associated costs) are available so you can track and control your MT spend
⚠️
A word of caution about automatic quality scores. Some TEnTs display automatic quality estimates alongside MT output—numerical scores that claim to indicate how good a machine-translated segment is. These scores should be interpreted with care. Most are based on metrics like BLEU (Bilingual Evaluation Understudy), which measure how closely the MT output matches a reference translation word for word. The problem is that BLEU and similar metrics reward literal, close-to-the-source translations—exactly the kind of output that often reads unnaturally in the target language. A segment can score well on BLEU while still sounding stiff, unidiomatic, or tonally wrong to a native reader. Treat these scores as a rough triage signal, not as a judgment of actual translation quality. Human review remains essential.
🔁

5. Translation Memory Design & Consistency Enforcement

Translation memory (TM) is one of the primary reasons to use a TEnT in the first place: previously translated segments are stored and reused, so translators don't start from zero every time similar or identical content appears. But TM quality depends heavily on who has contributed translations and how the tool manages matching, fuzzy thresholds, and context.

🌐 Why this matters for website localization Websites reuse content in ways that are not always obvious: boilerplate footer text, shared UI labels, recurring legal disclaimers, template-driven product descriptions. Good TM design means this content is translated once, stored, and reused consistently—not re-translated by a different team member who makes slightly different choices each time.

Criteria to Evaluate

  • The tool identifies which translator contributed each segment, making it easy to isolate and remove all segments from a specific provider if quality concerns arise about their work
  • The tool distinguishes between exact matches, context matches (same segment in the same surrounding content), and fuzzy matches—and presents each type differently to the translator
  • Context-aware matching is available: the tool considers surrounding segments when evaluating whether a match is appropriate, not just the segment text in isolation
  • Fuzzy match thresholds are configurable so you can decide what percentage of similarity triggers a suggestion versus requires translation from scratch
  • TM leverage reporting is available so you can predict costs, plan timelines, and communicate efficiently with clients or stakeholders
  • TMs can be shared across projects and across team members, so the whole team benefits from work done on earlier projects
  • TM entries can be edited, annotated, or flagged for review—so errors don't propagate indefinitely once they're in the memory
  • The tool supports import and export in TMX format (the standard TM interchange format) so you're not locked into a proprietary system
✅

6. Automated Quality Assurance

Automated QA runs checks across all translated segments before delivery, catching mechanical errors that human reviewers miss under time pressure: missing or extra spaces, untranslated segments, tag errors, terminology violations, number and date formatting inconsistencies, and more.

🌐 Why this matters for website localization Broken HTML tags or malformed placeholders in a localized file can corrupt a webpage or cause a CMS to reject the file entirely. Automated QA that checks tag integrity is especially important for web content. A missing closing tag in a translated string can take a whole section of a page down.

Criteria to Evaluate

  • Tag verification: the tool checks that all inline tags present in the source segment are correctly reproduced in the target
  • Terminology QA: the tool flags segments where a required term is missing or a forbidden term has been used
  • Consistency checks: identical or near-identical source segments are compared to verify that translation choices are consistent across the project
  • Number, date, and punctuation checks are configurable to match the conventions of the target locale
  • Untranslated segment detection flags any segment that was left in the source language
  • QA reports can be exported so issues can be triaged, assigned, and tracked to resolution
  • QA can be run automatically at defined workflow stages, not just manually on demand
🔌

7. CMS & Web Platform Integration

Manually exporting content from your website, uploading it to a TEnT, translating it, and then downloading and re-importing the translated files into the website is tedious, error-prone, and creates version control risks—especially on sites that update frequently. Direct integration between a TEnT and your web platform eliminates this friction by allowing content to flow automatically into and out of the translation workflow.

🌐 Why this matters for website localization The manual download-translate-upload cycle is one of the biggest sources of inefficiency and error in web localization. A CMS plugin or API connector means that when a page is updated in the source language, it can be automatically flagged for translation and routed to the right translator without anyone having to remember to do it. On the delivery side, approved translations can be published directly to the site without a developer having to touch a file. For teams managing ongoing content updates across multiple languages, this is not a convenience—it's a necessity.

Criteria to Evaluate

  • Native connector or plugin available for your CMS (WordPress, Shopify, HubSpot, or others you use)
  • Content can be pushed from the CMS to the TEnT automatically when new or updated pages are detected, without manual file export
  • Translated content can be published back to the CMS directly from the TEnT once approved, without requiring a file download and re-import
  • The connector supports incremental updates—only changed segments are sent for translation, not the entire page every time
  • When changed segments are sent for translation, the translator has access to the context in which the translation will appear
  • A public API is available for teams that use a custom-built platform not covered by pre-built connectors
  • The API is well-documented and actively maintained, with code samples available for common development environments
  • Webhooks or event-based triggers push content to your translation workflow automatically when source content changes
  • The integration preserves content structure and metadata (SEO fields, slug, alt text) so translatable elements beyond the main body text are not overlooked

🧮 Scoring Your Results

For each section, rate the tool on a scale of 1 to 5 for each criterion, then calculate an average for the section. Use the table below to record section scores for each tool you're comparing. Not all sections carry equal weight for every project—adjust the weighting column based on what matters most for your specific situation.

📋

Scoring Summary Table

How to score Rate each criterion: 1 = Not supported / Poor · 2 = Minimal / Buggy · 3 = Adequate · 4 = Good · 5 = Excellent / Fully supported. Calculate a section average, then multiply by your weight percentage. Sum the weighted scores for a final total out of 5.

Section Weights (suggested defaults)

  • Web File Format Support (15%) — adjust upward if you're working with complex web app files; downward if content is primarily HTML documents
  • Collaboration & Workflow (25%) — weight this heavily for distributed teams or multi-stage review processes
  • Terminology Management (20%) — weight this heavily if brand consistency and SEO keyword control are priorities
  • MT & LLM Integration Quality (15%) — weight this heavily if post-editing MT is a core part of your workflow
  • CMS & Web Platform Integration (15%) — weight this heavily if your site updates frequently or you're managing multiple languages on an ongoing basis
  • Translation Memory (5%) — weight this upward for projects with significant repeated content across many pages
  • Automated QA (5%) — a baseline requirement; most serious tools handle this adequately
