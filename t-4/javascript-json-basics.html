TOPIC 4

JavaScript & JSON Basics

Understand the fundamentals of JavaScript and JSON, and learn how externalizing translatable strings makes localization workflows more efficient and maintainable.

‚ö° What is JavaScript?

JavaScript (JS) is the programming language that brings websites to life. While HTML provides structure and CSS handles styling, JavaScript adds interactivity and dynamic behavior. Every time you click a button that opens a menu, submit a form, or see content update without a page reload, JavaScript is doing the work behind the scenes.

For localization professionals, understanding JavaScript matters because JS is how content that depends on user interaction is loaded dynamically on the page. Plus, in builds in which translatable strings haven't been properly externalized, a great deal of content that needs to be translated—such as error messages, button labels, quiz feedback, and dynamically loaded content—may actually live inside JavaScript files. If you only translate the HTML, you'll miss a significant portion of what users actually see and interact with.

Common JavaScript Elements
Variables — Named containers that store data. Think of them like labeled boxes. For example, let greeting = "Hello"; stores the text "Hello" in a variable called greeting.

Functions — Reusable blocks of code that perform a specific task. For example, a function called showWelcomeMessage() might display a greeting to the user in their language.

Control Structures — JavaScript uses conditional statements and loops to control the flow of the program. For example, an if statement might check the user's language preference and load the appropriate content.

Error Handling — JavaScript can catch and handle errors gracefully, which is important for providing a good user experience. For example, if a translation file fails to load, JavaScript can display a fallback message instead of leaving the user with a broken page.

üì¶ What is JSON?

JSON (JavaScript Object Notation) is a lightweight format for storing and exchanging structured data. It's human-readable, which makes it easy for translators to work with, and machine-readable, which makes it ideal for web applications. JSON has become a standard format for storing translatable content separately that is then loaded from application code, like JavaScript.

Common JSON Elements
Keys — Labels that identify each piece of data, always enclosed in double quotes. Example: "pageTitle". These labels are generally not translated, as they serve as identifiers for the application to retrieve the correct content.

Values — The actual data associated with a key. Values can be text strings, numbers, booleans (true/false), or more complex structures. Example: "Welcome to our site". Text strings are the most common type of content that gets translated in localization projects, ensuring that users can navigate a site in their preferred language.

Objects — Collections of key-value pairs wrapped in curly braces { }. Objects let you group related data together.

Arrays — Ordered lists of values wrapped in square brackets [ ]. Useful for storing lists of items like menu options or quiz answers.

A Simple JSON Example

Here's what a JSON file storing translatable content might look like. Notice how keys stay the same across languages while only the values change—this is the foundation of JSON-based localization. Also notice how each opening bracket must have an associated closing bracket.

üîó Externalizing Translatable Strings

Now that you understand the basics of JavaScript and JSON, let's explore one of the most important best practices in web localization: externalizing translatable strings. This means moving all user-facing text out of your code files (like JavaScript) and into separate data files (like JSON) that your application loads dynamically.

To illustrate this concept, we'll look at a refactored version of the Terminology Webpage you localized in Topic 3. You'll be able to compare the original and refactored versions side by side to see exactly what changed and why.

üí° What is Refactoring?
Refactoring means restructuring existing code to improve its organization without changing what it does. The webpage still looks and behaves exactly the same, but the underlying code is cleaner and easier to maintain.

Refactoring is one of the great advantages of working with code. You can start a project with a solid foundation, and then build upon it as your needs evolve. The Terminology Webpage started as a straightforward bilingual project. Now we're refactoring it to follow the best practice of externalized strings—making it easier to add new languages, update content, and collaborate with translators who may not be comfortable editing code files directly.

üîç The Problem: Embedded Strings

Remember the terminology-quiz_en-US.js file from the original Terminology Webpage project? In that file, all of the translatable strings—quiz questions, feedback messages, correct and incorrect responses—were embedded directly in the JavaScript code.

Original Approach: Strings Inside JavaScript

Here's a simplified view of how the quiz data was structured in the original JavaScript file. Notice how the translatable text is mixed in with the application logic: 

// terminology-quiz_en-US.js (original)
q1: {
    correct: "False",
    correctFeedback: "Correct! It's important to remember
        that just because communities of speakers use
        common language doesn't mean that those speakers
        share the exact same conceptualization...",
    incorrectFeedback: {
        "True": "Incorrect. It's important to remember..."
    }
},
q2: {
    correct: "use in a specific subject field",
    correctFeedback: "Correct! Special language is a natural
        language used in a subject field...",
    incorrectFeedback: {
        "independent of any specific subject field":
            "Incorrect. Special language is a natural
            language used in a subject field..."
    }
}

This approach creates several problems for localization:

  • Code duplication — To add Spanish, you had to duplicate the entire JavaScript file as terminology-quiz_es-MX.js, copying all the application logic along with the translatable text.
  • Maintenance burden — If you need to fix a bug in the quiz logic, you now have to update it in every language file separately.
  • Translator friction — Translators need to work inside code files, increasing the risk of accidentally breaking the application.
  • Scaling difficulty — Adding a third or fourth language means duplicating the entire file again each time.

‚ú® The Solution: Externalized Strings in JSON

In the refactored version, the translatable strings have been moved out of the JavaScript file and into separate JSON files—one for each language. The JavaScript file now contains only the application logic that the translator does not need to touch and dynamically loads the appropriate JSON file based on the user's selected language.

Refactored Approach: JSON Data Files + Shared Logic

// strings_en-US.json
{
    "q1": {
        "correct": "False",
        "correctFeedback": "Correct! It\u2019s important
            to remember that just because communities
            of speakers use common language doesn\u2019t
            mean that those speakers share the exact
            same conceptualization...",
        "incorrectFeedback": {
            "True": "Incorrect. It\u2019s important
                to remember..."
        }
    },
    "q2": {
        "correct": "use in a specific subject field",
        "correctFeedback": "Correct! Special language
            is a natural language used in a subject
            field...",
        "incorrectFeedback": {
            "independent of any specific subject field":
                "Incorrect. Special language is a
                natural language used in a subject
                field..."
        }
    }
}
// terminology-quiz.js (shared across all languages)
async function loadQuizContent(lang) {
    const response = await fetch(
        `strings_${lang}.json`
    );
    const quizData = await response.json();
    return quizData;
}

function checkAnswer(questionId, selectedAnswer) {
    const question = quizData[questionId];
    if (selectedAnswer === question.correct) {
        showFeedback(question.correctFeedback);
    } else {
        showFeedback(
            question.incorrectFeedback[selectedAnswer]
        );
    }
}
What Changed?
Before: Two separate JS files (terminology-quiz_en-US.js and terminology-quiz_es-MX.js), each containing both application logic and translatable text.

After: One shared terminology-quiz.js file with the application logic, plus separate JSON data files for each language (strings_en-US.json and strings_es-MX.json).

The result? Translators work with clean, structured JSON files instead of navigating complex code. Developers maintain logic in one place. Everyone benefits.

üí¨ Working with Quotes in JSON

One important detail to notice when working with JSON files: JSON uses double quotes (") to mark keys and values. This means that if your translatable text contains straight double quotes, they'll conflict with the JSON syntax and break the file.

How the Refactored Repo Handles Quotes

The refactored repository uses a combination of curly quotes (“ ”) and straight quotes (" ") in the JSON files. Curly quotes don't conflict with JSON syntax because JSON only recognizes straight double quotes as structural delimiters. 

// ‚ùå Broken: straight quotes inside a JSON string
{
    "feedback": "She said "hello" to everyone."
}
// The parser sees four separate fragments
// instead of one complete string!
// ‚úÖ Option A: Use curly quotes inside the string
{
    "feedback": "She said \u201chello\u201d to everyone."
}

// ‚úÖ Option B: Escape straight quotes with backslash
{
    "feedback": "She said \"hello\" to everyone."
}
Keeping Quotes Consistent
For a polished project, it's best to use one style of quotation marks consistently throughout your content. Some systems can automatically render straight quotes as curly quotes using CSS (with the font-variant-alternates property) or through JavaScript post-processing. This way, you can store simple escaped straight quotes in your JSON and let the presentation layer handle the visual styling.
