TOPIC 8

CMS & WordPress

Most websites you will localize are not built from scratch in a code editor. They run on a Content Management System. Here is what that means, and what localization looks like inside the most widely used one.

🗂️ What Is a Content Management System?

A Content Management System (CMS) is software that lets people build and manage a website without writing code by hand. Instead of editing raw HTML files, site owners work through a visual interface where they can write posts, upload images, change layouts, and manage settings.

Under the hood, a CMS still generates HTML, CSS, and sometimes JavaScript. But most of that happens automatically, based on templates and settings. The person running the site only sees a dashboard.

For a web localizer, this has real implications. The text you need to translate may not live in one clean HTML file. It can be spread across a database, theme files, plugin files, and configuration files, sometimes in formats you have not worked with before. Understanding the CMS is part of understanding the localization job.

🌐 WordPress: the big picture

WordPress powers approximately 43% of all websites globally, making it the most widely used CMS in the world. It comes in two very different forms.

WordPress.com vs. WordPress.org

WordPress.com is a hosted platform. You create an account and WordPress handles the servers, updates, and maintenance. It is the easiest way to get started and includes a free tier.

WordPress.org is the open-source software you download and install on your own web hosting server. It gives you full control over your site, including the ability to install any plugin or theme, but you are responsible for hosting, security, and updates. Most professional and business sites run on WordPress.org. When a client asks you to localize their WordPress site, it is likely a self-hosted WordPress.org installation.

In Topic 5, you built a site on WordPress.com to practice writing SEO-optimized content. That gave you a feel for the editing interface. In this topic, we zoom out to look at WordPress as a localization environment, including the parts that are less visible from the front end.

⚖️ WordPress for localizers: blessings and curses

WordPress makes some things easy and other things surprisingly complicated. Here is an honest look at both sides across the areas that matter most for localization.

The platform itself

  • Blessing: You can create as many free WordPress.com sites as you want, which makes it an accessible practice environment.
  • Curse: On WordPress.com, plugins require a paid plan, WordPress.com ads appear on free sites, and a custom domain also requires payment. Many of the tools most useful for localization, including translation plugins, are only available on paid plans or on self-hosted WordPress.org installations.

Themes

  • Blessing: Installing a theme instantly changes the look and layout of a site. There is no need to build a design from scratch.
  • Curse: Theme text strings, such as button labels, navigation items, and footer copy, are often not stored in the page content at all. They live in the theme's language files, typically .po and .pot files, which are accessed through the WordPress Admin Panel (wp-admin). To edit these, you need to know how to work within that format, and in some cases you will need to write HTML and CSS directly inside the translation interface to get the output you want.

    Additionally, some theme content lives in .php files, which are the template files that control how WordPress builds each page. A localizer working on a professional project will need to locate all of these files to make sure nothing gets missed. Identifying where all localizable strings actually live is a real part of the scoping process.

Styling

  • Blessing: Many themes offer pre-built style components. You add a block or a section and the design is already handled.
  • Curse: Where you go to customize styles depends on which type of theme is installed. Classic themes use Appearance > Customize, while block-based themes use the Site Editor, found under Appearance > Editor. These two interfaces work very differently from each other. Headers, footers, and automatically generated pages like blog archives or product listings may each be controlled in different places, and the setup changes from theme to theme. This inconsistency can make navigation genuinely difficult, even for experienced WordPress users. And as noted above, some style-related strings will still require going into the language files on the backend.

Plugins

  • Blessing: The WordPress plugin ecosystem is enormous. There is a plugin for almost any functionality you can imagine, including translation and localization workflows.
  • Curse: Full plugin functionality often requires payment. Plugins like WooCommerce (the most popular e-commerce plugin for WordPress) can also inject ads or promotional banners directly onto site pages. Getting a visually consistent result across a site that uses multiple plugins and a theme can be a significant challenge, since each plugin may bring its own styles.
The Gutenberg editor and the Code editor

Since WordPress 5.0 (released in December 2018), the default content editor is the block editor, known as Gutenberg. Content is built by adding and arranging blocks for text, images, buttons, and other elements. If you have used WordPress.com in this course, you have already worked in Gutenberg.

Within Gutenberg, there is a Code editor view accessible from the three-dot menu in the top-right corner of the editor. This lets you see and edit the raw block markup directly, which looks similar to HTML but uses WordPress-specific comment syntax to define blocks. When localizing content at the code level, this is where you would work.

🛠️ Localization tools for WordPress

Before any translation work begins, a localizer needs to identify all the places where text lives on a WordPress site. That means page content in the database, theme strings in .po/.pot files, template logic in .php files, and any strings added by plugins. Scoping a WordPress localization project thoroughly at the start will prevent gaps from showing up in the final product.

Once the scope is clear, there are several tool options for the translation work itself.

Loco Translate (free)
Loco Translate is the standard free plugin for translating WordPress themes and plugins directly from wp-admin. It gives you a browser-based interface for editing .po and .pot files without needing to download them or use a separate tool. It is a good starting point for understanding how WordPress language files work.
WPML (paid)
WPML (WordPress Multilingual Plugin) is the most widely used paid solution for building multilingual WordPress sites. It handles page content, theme strings, and plugin strings in one interface and supports multiple languages simultaneously. It is a professional-grade tool used in many commercial localization projects.
CAT tool connectors: a word of caution
Some plugins and services offer direct connectors between WordPress and CAT tools (Computer-Assisted Translation tools, also called TEnTs). While this sounds efficient, we do not recommend this approach for most website content. As covered in Topic 7, CAT tools segment text sentence by sentence. Website copy, including calls to action, landing page content, and blog posts, depends on tone, rhythm, and cultural adaptation to perform well. Sentence-level segmentation produces translations that are technically accurate but tonally flat, which damages both the user experience and SEO performance. CAT tools have their place in technical documentation and large structured content sets, but website copy deserves a more thoughtful approach.
