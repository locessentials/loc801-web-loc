TOPIC 9

Speedrun LQA Project

Perform localization quality assurance on a real multilingual FAQ page, focusing on completeness: making sure every translatable string has been translated.

🎯 Project Overview

In this project, you'll work with a simplified version of the a16z Speedrun FAQ page—a real startup program run by the venture capital firm Andreessen Horowitz. The project has already been set up with a full i18n system: every string on the page is tied to a translation key, and the Spanish translation file is partially complete. Your job is to finish it.

What makes this project distinctive is the built-in Translation Testing panel. You can toggle it on at any point during your work to see exactly how complete your translation is, highlight the strings that are still missing, and download your current JSON file. This is localization quality assurance with feedback built directly into the page.

🔍 Try It Before You Start

Before you fork anything, take the project for a spin. Navigate to locessentials.github.io/loc801-a16z-speedrun-faq/ in Chrome and press Ctrl+Shift+L to open the Translation Testing panel. Switch to Spanish and see what it looks like when a translation is incomplete. This panel is the heart of the project, so it pays toget familiar with it now.

Note: The testing panel works in Chrome. It does not work in Opera.

📁 How the Project Is Structured

Before you start translating, it's worth understanding how this project is organized and why. The README in the repository explains all of this in detail, so do be sure toread it. What follows here is a high-level orientation to help you find your footing.

One HTML File, Multiple Languages

Unlike earlier projects in this course, this one uses a single index.html for all languages rather than a separate HTML file per locale. This is a common pattern in modern web development. The HTML contains default English content as a fallback, and the data-i18n attributes throughout the page serve as lookup keys. When a user selects a language, the JavaScript swaps in the translated strings from the appropriate JSON file.

Where Your Work Lives

Almost all of your work happens in one file: locales/es.json. This is the Spanish translation template. Your job is to fill in the missing keys and values. The English reference file (locales/en.json) is your source. The locales/fr.json file is optional; you can create it if you wish.

How Attributes Are Localized

Some elements use not just data-i18n but also data-i18n-attr. This second attribute tells the localization script which HTML attribute to update, so things like placeholder, alt, and title. These aren't visible as text on the page, which makes them easy to miss. The testing panel will catch them for you.

📋 Your Tasks

The README outlines all required tasks. Here's a summary of what you should complete.

1

Complete the Spanish Translation

Using en.json as your reference, translate all strings in es.json. The file has around 2,000 words of content. Since the point here is just to get experience with the process, consider following an automatic translation + post-editing workflow so you can work more quickly.

  • Translate all interface strings: navigation, buttons, labels, placeholders
  • Translate all FAQ content, preserving the meaning and tone of the original
  • Pay attention to nested objects in the JSON structure; maintain the hierarchy exactly
  • Do not leave any key with an untranslated English value
2

Handle Special Content Correctly

Three categories of content require localization beyond simple text translation:

  • Dates: The application deadline (May 11th, 2025) should be formatted according to the Spanish locale convention
  • Currency: The investment amount ($1,000,000) should be formatted for a Spanish-speaking audience
  • Pluralization: Handle any plural forms that differ between English and Spanish
3

Verify Attributes Are Localized

Ensure that hidden attributes like placeholder, alt, title are correctly updated in your JSON file. These are easy to overlook but will show up as missing in the testing panel.

4

Test Your Work with the Panel

Use the Translation Testing panel (Ctrl+Shift+L) throughout your work to:

  • Check the percentage of translated strings at any point
  • Use "Highlight Missing" to visually identify what still needs attention
  • Toggle translation keys on to verify which strings map to which elements
  • Download your JSON file at any time to save your progress
5

Optional: Translate the Content into French

The fr.json file is an optional extension. If you'd like additional practice, apply the same process to add a French translation.

✍️ Reflection

After completing your translation, write a reflection addressing the three areas below, not a summary of what you did, but an analysis of the decisions you made and what you learned from them.

1. Translation Decisions and Challenges

What decisions did you make about terminology, tone, and register? How did you handle technical terms or startup-specific vocabulary that may not have a clean Spanish equivalent? Were there cases where you kept an English term (like "speedrun" or "a16z") rather than translating it, and why? Discuss at least two specific translation challenges and explain how you resolved them.

2. Locale Conventions

What locale conventions apply to your target locale (Spanish-speaking Mexico), and how do they differ from the English source? Were there any cases where the localization script handled formatting automatically, and any where you had to make a judgment call manually?

3. Using the Testing Panel as an LQA Tool

Reflect on the testing panel as an LQA tool. Did it catch anything that you might have missed on your own? Were there any limitations—things a completeness check can't tell you about translation progress and quality? How does this kind of automated checking relate to the broader LQA concepts covered in this topic?

💡 What Makes a Good Reflection
Strong reflections name specific examples, that is, specific strings, specific decisions, specific results. "The translation went smoothly" is not useful. Point to actual content from the FAQ, describe the actual problem you encountered, and connect your observations to what you've learned about LQA and web localization.
