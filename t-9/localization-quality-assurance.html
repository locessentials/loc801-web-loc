TOPIC 9

Localization Quality Assurance (LQA)

LQA is the review process that catches what translation alone doesn't: broken code, missing strings, inaccessible content, and discoverability gaps. Here's what to look for and how to look for it.

🎯 Learning Objectives

  • Understand what LQA is and why it is distinct from translation review
  • Identify the categories of errors most common in website and web application (and even game) localization
  • Apply a practical LQA sanity checklist to a localized website
  • Recognize quality issues related to code integrity, discoverability, and accessibility

🔍 What Is Localization Quality Assurance?

Localization Quality Assurance (LQA) is the structured review process that verifies a localized product works correctly in its target language and locale. It goes well beyond checking whether the translation reads naturally. LQA asks whether the entire localized experience is complete, functional, findable, and accessible.

In web localization, LQA means reviewing not just the body copy but every piece of text the site surfaces, including menus, error messages, browser tab titles, image alt texts, and form placeholders, as well as the underlying code that controls layout, behavior, and metadata. A missed string in a JSON file, a broken CSS class reference in a translated page, or a meta description left in the source language are all LQA failures, even if the translation itself is excellent.

LQA further extends to the user interface: button labels that get cut off when text expands in translation, date and number formats that don't match the target locale, and dialog boxes that overflow their containers. The website or web application has to be tested in the target language environment, not just read as a document.

Video content adds another layer of complexity. At a minimum, videos should include subtitles in the user's language that sync with audio, though a more fully localized experience would include custom video content as well.

⚙️ Why LQA Is Its Own Discipline

LQA is sometimes treated as a final proofreading pass: a translator gives the work one more read before it goes live. That's not LQA. It's a useful step, but it catches only linguistic errors that a reader would notice. It won't find a missing string in a navigation menu that's assembled dynamically from a JSON file. It won't catch a broken JavaScript function that prevents a form from submitting. It won't tell you whether your localized page is being indexed by search engines under the right language signals.

Genuine LQA requires someone who understands how the site is built, which content lives in which files, which behaviors are controlled by JavaScript, and which metadata elements affect discoverability, and who reviews the live rendered output rather than the source documents alone. For a small course site like this one, that may be you. For larger projects, it may be a team of dedicated LQA specialists or a developer working alongside the translator.

✅ LQA Sanity Checklist

The checklist below is organized from the most visible and straightforward items to the most technical. Work through it in order, as catching a completely untranslated page early saves you from reviewing it for everything else. Within each category, items are ordered from the simplest checks to the ones that require more technical investigation.
💡
Review the live rendered page, not just the source files. Many LQA errors, such as missing translations and broken layouts, only become visible once the page is assembled and loaded in a browser. When in doubt, open the page in your browser and use the developer tools (F12 in most browsers) to inspect what's actually being rendered.
🌐

Category 1 — Has Everything Been Translated?

Websites surface text from many different places, not just the main body copy, but structural elements assembled from separate files. A complete translation review means checking every place the site produces visible or functional text, not just the most obvious ones.

📁 A note on where content lives In a well-structured site, repetitive elements like navigation menus, headers, and footers are typically generated dynamically from JSON or PO files rather than hard-coded into each HTML page. This is good practice for maintainability, but it means that untranslated strings in those files will silently produce untranslated output across every page they appear on. Always check the source files as well as the rendered output.

What to Check

  • Page title and browser tab text — The <title> tag doesn't appear on the page itself, which makes it easy to forget. Check it in the page source or developer tools.
  • Headers and footers — Including the site name, tagline, navigation labels, social links, and any legal text like copyright notices or privacy policy links.
  • Menus and navigation — Both the main navigation and any secondary menus, dropdowns, or breadcrumb trails. In this course, these are generated from course-structure.json.
  • Main body content — All headings, body text, callout boxes, labels, and captions.
  • Form fields and placeholders — Input labels, placeholder text, submit button labels, and any helper text or field descriptions. These are often stored separately from the main content.
  • Error messages and validation text — Messages that appear when something goes wrong are frequently hardcoded outside the main content files and overlooked in translation.
  • Popup messages and system alerts — Any text that appears in modal dialogs, confirmation prompts, or dynamic notifications.
  • Images and video — Text embedded in images needs to be reproduced in a localized version; videos need localized subtitles or captions at minimum. (Image and video localization is not covered in depth in this course, but the omission should be a conscious decision.)
🔧

Category 2 — Code Integrity

Translation sometimes introduces code errors, particularly when translators are working inside HTML files and inadvertently modify tags, attributes, or references. The symptoms depend on which type of file was affected.

⚠️ How translation breaks code The most common source of code errors in localized files is a translator who modified text that was adjacent to a tag or attribute, or who worked in a tool that didn't correctly protect non-translatable elements. Always visually review the rendered page as well as the source.

What to Check

  • Character encoding — Confirm that <meta charset="UTF-8"> is declared and that no special characters (accented letters, quotation marks, em dashes) are rendering as garbled symbols. Encoding issues are particularly common when files are passed between tools or operating systems.
  • HTML errors — Broken or malformed HTML tags will manifest as raw code appearing visibly on the page, or as content that is missing or oddly positioned. Use your browser's developer tools to check for parse errors.
  • Broken internal links — Links that pointed to source-language pages may not have been updated to point to their localized equivalents. A Spanish page linking back to an English page is a common and easy-to-miss LQA error.
  • CSS errors — If class names or IDs were accidentally modified during translation, elements will lose their styling. Alternatively, some classes may require updates that were not possible to make in a CAT tool, where the code is hidden. Look for content that appears unstyled, misaligned, or visually broken compared to the source-language version.
  • JavaScript errors — Broken JavaScript typically manifests as functionality that silently stops working: a navigation menu that doesn't expand, a form that won't submit, or dynamic content that fails to load. Open the browser console to check for JavaScript errors.
📐

Category 3 — Layout and Locale Formatting

Translated text rarely takes up the same amount of space as the source. Spanish, for example, tends to run 20–30% longer than English. Layout review confirms that the translated content fits its container and that locale-specific formats like dates and numbers match the expectations of the target audience.

What to Check

  • Text expansion in UI elements — Check buttons, navigation labels, tab names, and any element with a fixed width or height. Translated text that overflows its container, truncates, or wraps unexpectedly is a layout failure, not a translation failure, and it's discovered during LQA.
  • Date, number, and currency formats — Confirm that any dates, numbers, and currency values displayed on the page follow the conventions of the target locale. For ESP-MEX, this means DD/MM/YYYY date format, a period as the thousands separator, and a comma as the decimal separator, the opposite of the ENG-USA convention.
♿

Category 4 — Accessibility

Accessibility review confirms that the localized site can be used by people who rely on assistive technology, screen readers, keyboard navigation, and other tools. Many accessibility attributes contain translatable text that is invisible to sighted users, making them particularly easy to overlook.

♿ Why this matters beyond compliance Accessibility failures in the localized version affect the same users who are most dependent on correct language signals: screen reader users whose software relies on the lang attribute to select the right voice, and keyboard users navigating with descriptive link text. A localized site that is less accessible than the source version has narrowed its audience.

What to Check

  • Image alt text — Confirm that alt attributes have been translated on all images. Alt text is read aloud by screen readers and also contributes to SEO. It is invisible on the rendered page, which makes it one of the most commonly skipped translation items.
  • ARIA labels and descriptions — Check that aria-label, aria-describedby, and similar attributes have been translated. These are used by assistive technology to describe interactive elements, such as buttons, icons, and form fields, that may not have visible text labels.
  • Link text — Review anchor text for links that were translated literally but are no longer descriptive. Phrases like "click here" or "read more" give screen reader users no information about where a link leads. Links should ideally describe their destination, which may require rewriting the surrounding sentence so that the linked text corresponds meaningfully to the URL it points to.
  • Color contrast — If any colors were adjusted during localization (for example, to accommodate a culturally different palette for the target market), verify that text maintains sufficient contrast against its background for users with low vision or color blindness.
🔎

Category 5 — Discoverability

A localized page that nobody can find is a missed opportunity. Discoverability review confirms that your translated content is correctly signaled to search engines and language models, and that the metadata actually matches what's on the page.

🌐 Covered in Topic 5 This course covered SEO fundamentals, keyword research, and LLM discoverability in Topic 5. The checks below apply those concepts to the localized version of the site.

What to Check

  • Language attribute — Confirm that the <html lang="..."> attribute matches the page language (e.g. lang="es" for Spanish, lang="en" for English). This is the most basic signal to search engines and assistive technology about what language the page is in.
  • Meta title and meta description — Open the developer tools and confirm that the <title> and <meta name="description"> tags have been translated and contain target-language keywords. A meta description left in the source language is one of the most common and consequential LQA oversights.
  • Keywords meta tag and Open Graph tags — Check that the <meta name="keywords">, og:title, and og:description tags have been localized. Open Graph tags control how the page appears when shared on social media.
  • Localized internal URLs — Confirm that the page's URL has been localized where appropriate. A localized page served at an English URL (e.g. /t-9/localization-quality-assurance.html instead of /t-9/aseguramiento-de-calidad.html) sends mixed signals to search engines and looks unprofessional to users.
  • External links and surrounding copy — Verify that external links point to localized versions of the linked resource where they exist (e.g. a link to an English Wikipedia article should point to the Spanish equivalent), or to an entirely new resource if a localized version of the original link is not available. Check that the anchor text and any surrounding copy have been updated accordingly. A sentence written around a source text link may read awkwardly or be factually inconsistent once the link destination changes.
  • Keyword frequency in body copy — Verify that target-language keywords (or synonyms) appear with reasonable frequency in the body text. A translated page that doesn't write around the keywords in its metadata misses an opportunity to reinforce topical relevance. Google can recognize synonyms, and body copy that naturally reflects the page's target keywords sends a stronger signal.
  • Keyword conflicts between pages — If multiple pages on the site are targeting the same keyword in the target language, they will compete against each other in search results. Each page should have a distinct keyword focus.
  • hreflang attributes — For a bilingual site, confirm that each page correctly declares its language alternates using hreflang tags. These tell search engines which pages are translations of each other and prevent them from being indexed as duplicate content.
  • Writing quality and LLM discoverability — If a CAT tool was used, review the body copy for the telltale signs of segment-by-segment translation: abrupt sentence transitions, unnatural repetition, and phrasing that is technically correct but reads mechanically. Poor writing quality degrades performance with both traditional search engines and LLM-based discovery tools. Passages like this may need to be rewritten rather than corrected word by word.

📌 LQA as a Habit, Not a Handoff

The most effective LQA happens throughout a localization project, not just at the end. Catching a missing string in a JSON file before translation is complete costs very little. Catching it after the site is live when a user screenshots your English menu in a Spanish-language page and shares it online costs considerably more.

For a project of the scale you'll be working with in this course, a single pass through this checklist before publishing each localized page is sufficient. For larger projects, such as a full e-commerce site or a multilingual SaaS product, LQA becomes a formal stage with dedicated reviewers, issue tracking, and defined acceptance criteria. The principles, however, are the same.
