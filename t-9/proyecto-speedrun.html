TEMA 9

Proyecto de Speedrun

Realiza control de calidad lingüístico en una página FAQ multilingüe real, con foco en la integridad: asegurarte de que cada cadena traducible haya sido traducida.

🎯 Resumen del proyecto

En este proyecto trabajarás con una versión localizada de la página FAQ de a16z Speedrun, un programa real para startups del fondo de capital de riesgo Andreessen Horowitz. El proyecto ya viene configurado con un sistema i18n completo: cada cadena de texto en la página está vinculada a una clave de traducción, y el archivo de traducción al español está parcialmente completo. Tu tarea es terminarlo.

Lo que hace especial a este proyecto es el panel de pruebas de traducción que tiene integrado. Puedes activarlo en cualquier momento para ver exactamente qué tan completa está tu traducción, resaltar las cadenas que aún faltan y descargar tu archivo JSON con el progreso actual. Así es el control de calidad de localización con retroalimentación directamente en la página.

🔍 Pruébalo antes de empezar

Antes de hacer un fork de nada, explora el proyecto. Abre locessentials.github.io/loc801-a16z-speedrun-faq/ en Chrome y presiona Ctrl+Shift+L para abrir el panel de pruebas de traducción. Cambia al español y observa cómo se ve la página cuando una traducción está incompleta. Este panel es el núcleo del proyecto, entonces familiarízate con él desde ahora.

Nota: El panel de pruebas funciona en Chrome. No funciona en Opera.

📁 Cómo está organizado el proyecto

Antes de empezar a traducir, vale la pena entender cómo está organizado este proyecto y por qué. El README del repositorio explica todo esto en detalle, entonces léelo. Lo que encontrarás aquí es una orientación general para que puedas ubicarte.

Un solo archivo HTML, varios idiomas

A diferencia de proyectos anteriores del curso, este usa un solo index.html para todos los idiomas, en lugar de un archivo HTML separado por idioma. Este es un patrón común en el desarrollo web moderno. El HTML contiene el contenido en inglés como respaldo, y los atributos data-i18n distribuidos por toda la página funcionan como claves de búsqueda. Cuando alguien elige un idioma, el JavaScript reemplaza las cadenas de texto con las traducciones del archivo JSON correspondiente.

Dónde vive tu trabajo

Casi todo tu trabajo ocurre en un solo archivo: locales/es.json. Esta es la plantilla de traducción al español. Tu tarea es añadir los claves ycompletar los valores. El archivo de referencia en inglés (locales/en.json) es tu fuente. El archivo locales/fr.json es opcional; puedes crearlo si quieres.

Cómo se localizan los atributos

Algunos elementos usan no solo data-i18n sino también data-i18n-attr. Este segundo atributo le indica al script de localización qué atributo HTML debe actualizar, cosas como placeholder, alt y title. Estos atributos no son visibles como texto en la página, lo que hace fácil pasarlos por alto. El panel de pruebas los detectará por ti.

📋 Tus tareas

El README detalla todas las tareas requeridas. Aquí tienes un resumen de lo que debes completar.

1

Completa la traducción al español

Usando en.json como referencia, traduce todas las cadenas en es.json. El archivo contiene alrededor de 2,000 palabras de contenido. Como en punto de este proyecto es practicar el proceso, considera seguir un flujo de trabajo de traducción automática + posedición para trabajar con mayor velocidad.

  • Traduce todas las cadenas de interfaz: navegación, botones, etiquetas, marcadores de posición
  • Traduce todo el contenido del FAQ, preservando el significado y el tono del original
  • Presta atención a los objetos anidados en la estructura JSON, mantén la jerarquía exactamente igual
  • No dejes ninguna clave con un valor en inglés sin traducir
2

Maneja correctamente el contenido especial

Hay tres categorías de contenido que requieren localización más allá de la simple traducción de texto:

  • Fechas: La fecha límite de solicitud (May 11, 2025) debe estar formateada según las convenciones del español de México
  • Moneda: El monto de inversión ($1,000,000) debe formatearse para una audiencia hispanohablante
  • Pluralización: Resuelve las formas plurales que difieren entre el inglés y el español
3

Verifica que los atributos estén localizados

Asegúrate de que los atributos no visibles, como placeholder, alt y title, estén correctamente actualizados en tu archivo JSON. Son fáciles de pasar por alto, pero el panel de pruebas los detectará.

4

Prueba tu trabajo con el panel

Usa el panel de pruebas de traducción (Ctrl+Shift+L) a lo largo de tu trabajo para:

  • Revisar el porcentaje de cadenas traducidas en cualquier momento
  • Usar "Resaltar faltantes" para identificar visualmente lo que aún necesita atención
  • Activar las claves de traducción para verificar qué cadenas corresponden a qué elementos
  • Descargar tu archivo JSON en cualquier momento para guardar tu progreso
5

Opcional: Traduce el contenido en francés

El archivo fr.json es una extensión opcional. Si quieres practicar más, aplica el mismo proceso para añadir la traducción en francés.

✍️ Reflexión

Después de completar tu traducción, escribe una reflexión que aborde las tres áreas siguientes. Esto no es un resumen de lo que hiciste, es un análisis de las decisiones que tomaste y lo que aprendiste de ellas.

1. Decisiones y retos de traducción

¿Qué decisiones tomaste sobre terminología, tono y registro? ¿Cómo manejaste los términos técnicos o el vocabulario propio del ecosistema de startups que no tenía un equivalente claro en español? ¿Hubo casos en los que preferiste mantener un término en inglés (como "speedrun" o "a16z") en lugar de traducirlo, y por qué? Analiza al menos dos retos concretos de traducción y explica cómo los resolviste.

2. Convenciones locales

¿Qué convenciones locales aplican para tu variante de destino (español de México) y en qué se diferencian del inglés fuente? ¿Hubo casos donde el script de localización gestionó el formato automáticamente, y otros donde tuviste que tomar una decisión manualmente?

3. El panel de pruebas como herramienta de control de calidad

Reflexiona sobre el panel de pruebas como herramienta de LQA. ¿Detectó algo que quizás habrías pasado por alto por tu cuenta? ¿Tiene limitaciones, cosas que una verificación de integridad no puede decirte sobre el progreso y la calidad de la traducción? ¿Cómo se relaciona este tipo de verificación automatizada con los conceptos de LQA que se abordaron en este tema?

💡 Qué hace una buena reflexión
Las reflexiones sólidas citan ejemplos concretos: cadenas específicas, decisiones específicas, resultados específicos. "La traducción fue fluida" no es una respuesta útil. Señala contenido real del FAQ, describe el problema real que encontraste y conecta tus observaciones con lo que has aprendido sobre LQA y localización web.
